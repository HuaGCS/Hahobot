# Repository Guidelines

## Project Structure & Module Organization
`hahobot/` is the main Python package. Core agent logic lives in `hahobot/agent/`, channel integrations in `hahobot/channels/`, providers in `hahobot/providers/`, and CLI/config code in `hahobot/cli/` and `hahobot/config/`. Localized command/help text lives in `hahobot/locales/`. Bundled prompts and built-in skills live in `hahobot/templates/` and `hahobot/skills/`, while workspace-installed skills are loaded from `<workspace>/skills/`. Tests go in `tests/` with `test_<feature>.py` names. The WhatsApp bridge is a separate TypeScript project in `bridge/`.

## Build, Test, and Development Commands
- `uv sync --extra dev`: install Python runtime and developer dependencies from `pyproject.toml` and `uv.lock`.
- `uv run pytest`: run the full Python test suite.
- `uv run pytest tests/test_web_tools.py -q`: run one focused test file during iteration.
- `uv run pytest tests/test_skill_commands.py -q`: run the ClawHub slash-command regression tests.
- `uv run ruff check .`: lint Python code and normalize import ordering.
- `uv run hahobot agent`: start the local CLI agent.
- `uv run hahobot sessions list -c-json`: inspect recent saved sessions in the active workspace.
- `uv run hahobot sessions export cli:direct --format md`: export one saved session to a local artifact.
- `uv run hahobot sessions compact cli:direct --json`: manually run session token consolidation for one saved session.
- `uv run hahobot memory index rebuild --json`: rebuild the optional SQLite FTS archive index from JSON sidecars.
- `uv run hahobot repo status --json`: inspect the active workspace's local Git state without modifying the repo.
- `uv run hahobot review --staged`: review the active workspace's staged diff with the configured model.
- `uv run hahobot doctor --json`: inspect runtime readiness without writing files.
- `uv run hahobot agent --multiline`: start interactive CLI chat with multiline input enabled.
- `uv run hahobot agent --pick-session`: choose a recent CLI session interactively before chatting.
- `cd bridge && npm install && npm run build`: install and compile the WhatsApp bridge.
- `bash tests/test_docker.sh`: smoke-test the Docker image and onboarding flow.

## Coding Style & Naming Conventions
Target Python 3.11+ and keep Python code consistent with Ruff: 4-space indentation, `snake_case` for functions/modules, `PascalCase` for classes, and `UPPER_SNAKE_CASE` for constants. Ruff uses a 100-character target; stay near it even though long-line errors are ignored. Prefer explicit type hints and small functions. In `bridge/src/`, keep the current ESM TypeScript style and avoid reformatting unrelated lines.

## Testing Guidelines
Write pytest tests using `tests/test_<feature>.py` naming. Add a regression test for every bug fix and cover async flows, channel adapters, and tool behavior when touched. If you change slash commands or command help, update the related loop/localization tests and, when relevant, Telegram command-menu coverage. `pytest-asyncio` is already enabled with automatic asyncio handling. There is no published coverage gate, so prefer targeted assertions over smoke-only tests.

## Commit & Pull Request Guidelines
Recent history favors short Conventional Commit subjects such as `fix(memory): ...`, `feat(web): ...`, and `docs: ...`. Use imperative mood, add a scope when it helps, and keep unrelated changes out of the same commit. PRs should summarize the behavior change, note config or channel impact, list the tests you ran, and link the relevant issue or PR discussion. Include screenshots only when CLI output or user-visible behavior changed.
- When you manually port or intentionally skip behavior from a tracked upstream (`HKUDS/nanobot`, `lsdefine/GenericAgent`, `thedotmack/claude-mem`, `Dataojitori/nocturne_memory`, or `openJiuwen/jiuwenswarm` at https://atomgit.com/openJiuwen/jiuwenswarm), update `UPSTREAM_PARITY.md` in the same patch so upstream sync state does not live only in commit history. `nanobot` is the behavior-parity target; the others are architectural/ideas upstreams mapped onto existing hahobot surfaces.

## Security & Configuration Tips
Do not commit real API keys, tokens, chat logs, or workspace data. Keep local secrets in `~/.hahobot/config.json` and use sanitized examples in docs and tests. If you change authentication, network access, or other safety-sensitive behavior, update `README.md` or `SECURITY.md` in the same PR.
- If a change affects user-visible behavior, commands, workflows, or contributor conventions, update both `README.md`, `README_ZH.md` and `AGENTS.md` in the same patch so runtime docs and repo rules stay aligned.
- Startup config loading should auto-normalize compatible legacy config shapes. When no explicit config path is provided, check `~/.hahobot/config.json` first and only fall back to `~/.nanobot/config.json` if the hahobot config is absent. In that fallback case, copy the legacy config into `~/.hahobot/config.json` instead of treating it as an in-place migration; when the legacy default workspace still exists, preserve it by pinning `agents.defaults.workspace` instead of silently orphaning user data. Deprecated keys such as `agents.defaults.memoryWindow` and legacy locations such as `tools.exec.restrictToWorkspace` should be rewritten to the canonical current schema during load.
- Keep legacy `nanobot` automation working during the rename transition. The packaged app should continue exposing a `nanobot` CLI entrypoint, `python -m nanobot`, and top-level SDK imports such as `from nanobot import Nanobot`; admin auth/language cookies should prefer `hahobot_*` names while still accepting and cleaning up the legacy `nanobot_*` cookies.
- `gateway.admin` controls the built-in per-instance admin page. Keep it disabled by default unless explicitly configured, require a non-empty `authKey` when enabled, and treat `/admin` as unreachable when the switch is off.
- The built-in admin shell may absorb Hermes-style dashboard layouts, but keep it in the same aiohttp/Jinja runtime instead of introducing a second standalone SPA just for local ops pages.
- `gateway.status` controls the optional HTTP status endpoint for Star-Office-UI-style dashboards. Keep it disabled by default, serve it from the same gateway process at `/status`, return `404` when off, require `Authorization: Bearer <authKey>` when `authKey` is configured, keep JSON stable for script/dashboard clients, and allow browser requests to render a human-readable status page for the same runtime state, including the recent persona's `PROFILE.md` / `INSIGHTS.md` memory-layer summary.
- `gateway.status.push` may optionally push status directly to Star-Office-UI over HTTP. Keep `mode=guest` as the default using `join-agent` / `agent-push` with a required `joinKey`, and support `mode=main` for driving the built-in main `/set_state` contract without a `joinKey`. Treat push settings as hot-reloadable runtime config.
- The built-in admin page edits the active instance config plus the started process's runtime workspace/persona files. Do not silently retarget it to some other workspace outside the current instance context.
- The persona detail page may expose a structured metadata summary for `PROFILE.md` / `INSIGHTS.md`. Keep it read-only, locale-backed, and based on the same `<!-- hahobot-meta: confidence=... last_verified=YYYY-MM-DD -->` contract that Dream uses; include leftover legacy `(verify)` counts for migration visibility.
- The persona detail page may expose a migration preview/action for older profile-style `USER.md` content. Keep it local and deterministic: preview the exact post-migration `USER.md` / `PROFILE.md` / `INSIGHTS.md` content and move counts before writing, migrate obvious user-profile sections into `PROFILE.md`, workflow/collaboration guidance into `INSIGHTS.md`, preserve existing target content, and leave unrelated relationship-framing content in `USER.md`.
- The built-in admin page should also expose a slash-command reference page for the currently supported chat commands, including aliases and concrete usage examples.
- The built-in admin page may also expose read-only Hermes-style workspace dashboards for saved sessions, loaded skills, and persisted cron jobs. Keep them locale-backed, scoped to the active runtime workspace, and non-mutating unless an explicit write flow is added later.
- The admin slash-command reference should use a split layout: command list on the left, selected-command details on the right, and stack vertically on narrow screens.
- The persona detail editor should include locale-backed inline explanations for each editable file block, not just the raw filename.
- The persona detail editor may also expose a visual companion-scene section backed by `.hahobot/st_manifest.json`. Keep common `/scene` fields such as `reference_image`, `reference_images`, `scene_prompts`, `scene_captions`, and `response_filter_tags` editable without requiring raw JSON, while preserving a raw manifest editor as the fallback for unknown fields.
- The persona detail editor may also expose a scene-preview action that generates a one-off `/scene` result for the current persona. Keep it local to the active runtime workspace/config, reuse the current `tools.imageGen` settings, and write generated output under `workspace/out/image_gen`.
- The persona detail editor may also let operators save the current `/scene` preview back into persona `scene_prompts` / `scene_captions` as a named template. Preserve unknown manifest fields and treat that flow as a deterministic editor over `.hahobot/st_manifest.json`, not a separate scene store.
- Admin UI strings must come from `hahobot/locales/en.json` and `hahobot/locales/zh.json`, not hardcoded dictionaries in `hahobot/gateway/admin.py`. Keep Chinese as the default admin locale, allow English switching, and preserve the visual config editor plus raw JSON fallback.
- Visual admin config fields should include locale-backed hover descriptions for every exposed option, so operators can inspect field semantics without leaving the page.
- Admin config saves should force-reload hot-reloadable runtime settings for the current instance where supported, and every visual field should be marked directly in the UI as either hot-reloadable or restart-required.
- The admin visual config should expose `gateway.status.enabled` / `gateway.status.authKey` plus `gateway.status.push.*` for Star-Office-UI integration, `tools.exec.*` for shell-tool runtime control, channel runtime controls such as `channels.transcriptionProvider` / `channels.transcriptionLanguage`, and common single-instance channel credential cards for `whatsapp`, `telegram`, `discord`, `feishu`, `dingtalk`, `slack`, `qq`, `matrix`, `weixin`, and `wecom`. If a channel is already using `instances`, keep it read-only in the visual editor and preserve raw JSON as the edit path.
- The built-in admin page may also expose a dedicated Weixin QR-login helper for the current instance. Keep it scoped to the active `channels.weixin` config/state file, and make the UI clear that a non-empty `channels.weixin.token` in `config.json` still takes precedence over saved QR-login state at runtime.
- The admin visual config should expose a dedicated Memorix MCP section for `tools.mcpServers.memorix` with locale-backed labels/tooltips and hot-reload badges; leave generic multi-server MCP editing in the raw JSON fallback.

## Chat Commands & Skills
- Slash commands are handled in `hahobot/agent/loop.py`; keep parsing logic there instead of scattering command behavior across channels.
- When a slash command changes user-visible wording, update both `hahobot/locales/en.json` and `hahobot/locales/zh.json`.
- If a slash command should appear in Telegram's native command menu, also update `hahobot/channels/telegram.py`.
- `channels.telegram.streamEditInterval` is the configurable throttle for Telegram streamed reply edits. Keep it wired through schema/config rather than hardcoded channel constants, and treat it as a safe hot-reloadable runtime field.
- `channels.telegram.inlineKeyboards` may render `OutboundMessage.buttons` as Telegram native inline keyboards. Keep callback payloads under Telegram's 64-byte cap and preserve button-label text fallback when native keyboards are disabled.
- `channels.discord.streaming` should stay enabled by default, and the common Discord UX fields `readReceiptEmoji`, `workingEmoji`, and `workingEmojiDelay` should remain available in schema/docs/admin surfaces together.
- `channels.discord.proxy` with optional `proxyUsername` / `proxyPassword` should remain available in schema/docs/admin surfaces; if only one credential half is set, ignore proxy auth and warn rather than half-configuring it.
- Feishu topic/thread replies should keep every split outbound part on the Reply API path using the root/message id, not only the first part.
- The email channel is non-streaming: `EmailChannel.send()` must skip `_progress` messages (progress text and tool hints) instead of emailing one per update, otherwise enabling `channels.sendProgress` / `channels.sendToolHints` would send a near-empty email after each tool call. Keep every channel's send path progress-aware.
- Outbound email attaches `OutboundMessage.media` files via `EmailChannel._attach_media` (mimetype-guessed), bounded by `channels.email.maxAttachmentsPerEmail` and `channels.email.maxAttachmentSize` (shared with inbound). Skip missing/oversized files with a warning rather than failing the send, so the message body still goes out.
- `channels.websocket` may expose the runtime as a local WebSocket server. Keep `host` local by default, support tokenless local use plus optional `tokenIssuePath` / `tokenIssueSecret`, and preserve the simple `ready` / `message` / `delta` / `stream_end` frame contract.
- `/stchar` provides `list`, `show`, and `load` as NanoMate-style aliases over workspace personas. Keep it routed through the same session persona switching logic as `/persona set`.
- `/persona set` (and the `/stchar load` alias) must not block the switch confirmation on the outgoing persona's memory archival. Archiving the unconsolidated tail can make a slow LLM call, so deep-copy a detached snapshot of the session that still carries the OLD persona metadata, hand archival + memory flush to a background task, and switch/clear/save/reply in the foreground. The detached copy is what pins the previous persona's store (`_get_store`/`_get_archive_store` resolve from `session.metadata["persona"]`), so never archive against the freshly switched live session. The archive path's raw-dump fallback makes backgrounding safe; on background archival failure or incompletion, log it and push a localized `memory_archival_failed_persona` notice to the originating channel rather than silently dropping the tail. This is a deliberate change from the older block-then-abort behavior — the switch always succeeds now.
- `/preset` currently exposes read-only preset inspection for the active or named persona, based on `STYLE.md` plus `.hahobot/st_preset.json`.
- `/scene` provides `list`, `daily`, `comfort`, `date`, `generate <brief>`, and persona-defined custom scene names as direct companion-image shortcuts. Keep it wired through the built-in `image_gen` tool, prefer persona reference images when available, and return a media-bearing chat response instead of a prompt-only stub.
- Persona-local `.hahobot/st_manifest.json` may also define `scene_prompts` / `scene_captions` (and camelCase aliases) for `/scene` overrides. Treat them as deterministic prompt/caption supplements layered on top of the built-in scene shortcuts; keys from those maps and from `reference_images` should become valid `/scene <name>` custom shortcuts.
- `/skill` currently supports `search`, `install`, `uninstall`, `list`, `update`, `derive`, `supersede`, and `lint`. Keep subcommand dispatch in `hahobot/agent/loop.py`.
- Bundled workflow skills now include always-on `workflow-core` plus opt-in `plan`, `verify`, and `skill-derive`; keep them aligned with the current hahobot runtime rather than GenericAgent-specific assumptions. Plans/TODOs for future autonomous work should be treated as drafts until independently reviewed, preferably by `spawn(..., mode="verify")`; do not use self-review as permission to execute the plan.
- `agents.defaults.disabledSkills` may hide specific bundled or workspace skills from both the main agent skill summary and subagent skill summaries. Treat values as skill directory names, and keep the filter applied to always-on skills as well.
- Skill frontmatter `metadata` may carry hahobot-local lifecycle hints such as `triggers`, `tool_tags`, `supersedes`, `last_used`, and `success_count`; keep parsing compatible with current `hahobot` / legacy `nanobot` / `openclaw` wrappers.
- When runtime `read_file` actually opens `workspace/skills/<name>/SKILL.md`, best-effort write back `last_used`, and increment `success_count` only for turns that finish without `error`, `tool_error`, `empty_final_response`, or `max_iterations`.
- Runtime skill summaries for the main agent and subagents should stay query-aware and top-k scoped instead of dumping every visible skill into every prompt. When a newer skill declares `supersedes`, hide the superseded target from the shared runtime summary if the newer skill is available.
- `/skill supersede <newer> <older> [more...]` is the explicit local edit path for `supersedes`. `remove` and `clear` variants should cover the subtractive side as well. Keep all of these limited to workspace skills, validate add-target names against currently discoverable skills, and do not treat them as an auto-delete or auto-merge workflow.
- `/skill lint` is the read-only hygiene surface for local skill growth. Keep it reporting missing `supersedes` targets plus likely overlap among still-visible skills; do not auto-merge or auto-delete skills from lint alone.
- The `spawn` tool now supports `mode=explore|implement|verify`; enforce the mode boundary through the subagent tool registry rather than prompt text alone.
- The built-in `self_inspect` tool is a read-only runtime snapshot surface for the main agent. Keep it JSON-oriented, bound to the actual session key, and limited to inspection of runtime/session/tool/subagent state rather than self-modification.
- The built-in `notebook_edit` tool should stay constrained to `.ipynb` cell replace/insert/delete flows, may create a new notebook only on `insert`, and should be registered for the main agent plus `spawn(mode=implement)` workers only.
- Subagent completion announcements should carry `subagent_task_id` metadata and be persisted into session history before prompt assembly, with task-id dedupe so crash recovery keeps the follow-up durable without injecting the same content twice in one prompt.
- `/mcp` supports the default `list` behavior (and explicit `/mcp list`) to show configured MCP servers and registered MCP tools.
- `/update` should support `/update`, `/update check`, `/update force`, and `/update bridge`. The default mode syncs the current Git checkout with `git pull --ff-only`, runs `uv sync --locked --all-extras`, refreshes the local WhatsApp bridge when `channels.whatsapp` is enabled, and then restarts the current process. `check` is dry-run only, `force` skips the clean-worktree precheck without adding destructive Git behavior, and `bridge` only refreshes the WhatsApp bridge then restarts. Keep all modes chat-routable and report failures back to the same chat without restarting.
- `tools.web.search.provider` currently supports `brave`, `searxng`, and `duckduckgo`. Keep Brave API-key validation and SearXNG base-URL validation explicit, and run DuckDuckGo searches as exclusive tool calls so concurrent tool batches do not group multiple DuckDuckGo lookups together.
- `WebFetchTool` must validate every redirect hop's resolved IP against the SSRF policy before issuing the next request (walk redirect chains manually with `follow_redirects=False`, e.g. through `_get_with_safe_redirects`). Validating only the final URL after httpx silently followed the chain is not sufficient because intermediate hops may transit disallowed targets.
- MCP `sse` / `streamableHttp` server URLs are operator-configured in `config.json`, so the configured host is **trusted even when it is loopback/LAN** — local MCP servers (e.g. `http://127.0.0.1:3211/mcp`) are the common case and must keep working. Do not block the configured `cfg.url` itself. The only SSRF vector to guard is a configured-public server that *redirects* to an internal address: both transport httpx clients carry a `_make_mcp_redirect_validator(cfg.url)` request event hook (these transports keep `follow_redirects=True`) that allows requests to the configured host and rejects only a redirect to a *different* host resolving to a private/internal address (`validate_resolved_url`). Keep this redirect guard when touching MCP transport setup; do not regress it into a blanket private-address block.
- `tools.mcpServers.<name>.enabledTools` may filter one MCP server down to a subset of raw MCP tool names or wrapped hahobot tool names. Keep `["*"]` as the default meaning "all tools", treat `[]` as "register none", and surface unmatched names as warnings instead of silently ignoring the mismatch.
- Stdio MCP servers must not pollute the interactive CLI with their stderr: `_open_session` routes each stdio subprocess's stderr through `_stderr_to_logger` (an `os.pipe` whose read end a daemon thread drains into `logger.debug("[mcp:<name>] …")`), and closes the parent write end after spawn so the reader sees EOF on exit. Keep this for stdio transport only (sse/streamableHttp spawn no subprocess), and keep it best-effort — on any setup failure fall back to the SDK default (`sys.stderr`) rather than blocking MCP startup.
- MCP tool/resource/prompt wrappers auto-reconnect a terminated session before falling back to their error-string result. Keep transport setup in the shared `_open_session(name, cfg, stack)` helper (used by both initial connect and reconnect), gate reconnect through `_MCPWrapperBase._refresh_session_after_termination` so it fires at most once per `execute()`, and route rebuilds through the per-server `_MCPServerConnection` coordinator (lock + generation counter) so concurrent terminated calls trigger exactly one rebuild. Detection lives in `_is_session_terminated` ("session terminated"/"connection closed"); preserve the existing `TimeoutError` / `CancelledError` (re-raise only on external cancel) / `McpError` handling untouched.
- `/status` should return plain-text runtime info plus the active session persona's memory-layer summary, and stay wired into `/help` plus Telegram's command menu/localization coverage; the browser `/status` page should also show the recent task's current step / next step summary and the recent persona's `PROFILE.md` / `INSIGHTS.md` summary.
- `hahobot serve` starts the built-in OpenAI-compatible API. Keep it local-bind by default via `api.host` / `api.port`, reuse the fixed session `api:default`, require exactly one `user` message per request, accept both JSON and `multipart/form-data` request bodies, support text-first extraction for inline base64 file blocks and uploaded text-like files, emit stable placeholders for binary/image attachments on the direct API path, and keep `stream=true` unsupported unless the API contract is deliberately expanded.
- Agent runtime config should be hot-reloaded from the active `config.json` for safe in-process fields such as `tools.mcpServers`, `tools.web.*`, `tools.exec.*`, `tools.imageGen.*`, `tools.restrictToWorkspace`, `agents.defaults.workspace`, `agents.defaults.model`, `agents.defaults.maxToolIterations`, `agents.defaults.contextWindowTokens`, `agents.defaults.maxTokens`, `agents.defaults.temperature`, `agents.defaults.reasoningEffort`, `agents.defaults.toolHintMaxLength`, `agents.defaults.timezone`, `agents.defaults.disabledSkills`, `agents.defaults.idleCompactAfterMinutes`, `channels.sendProgress`, `channels.sendToolHints`, `channels.sendMaxRetries`, `channels.transcriptionProvider`, `channels.transcriptionLanguage`, `channels.voiceReply.*`, `memory.user.backend`, `memory.user.sqlite.*`, and `memory.archive.indexBackend`. Channel connection settings and provider credentials still require a restart.
- `agents.defaults.unifiedSession` may route all non-overridden inbound surfaces through one shared session key (`unified:default`). Treat it as a restart-required routing change, and never let it overwrite an explicit thread/session override from a channel integration.
- `agents.defaults.idleCompactAfterMinutes` controls idle-session auto compact. Keep it compatible with the legacy `sessionTtlMinutes` alias, archive stale live prefixes through the existing memory/history path, preserve a recent legal live suffix, and inject a one-shot resume summary back into runtime context when the session wakes up again.
- `agents.defaults.dream` backs the built-in Dream system job that `hahobot gateway` registers on startup. Keep `intervalH` as the canonical schedule field, accept legacy `cron` / `model` input for compatibility, and treat Dream config changes as restart-required rather than hot-reloadable.
- The workspace-scoped cron scheduler should periodically wake and reload `cron/jobs.json` even when the next known job is far in the future or absent, so external job additions/edits remain visible without a gateway restart.
- `gateway.cron.maxSleepMs` controls that periodic wake interval and should remain hot-reloadable through the running gateway/admin flow.
- `agents.defaults.providerPool` may define an ordered list of provider targets for `failover` or `round_robin` routing. Keep credentials under `providers.*`, let `providerPool` take precedence over `agents.defaults.provider` when non-empty, and treat pool changes as restart-required routing changes.
- Provider error surfaces should preserve nested transport causes such as DNS failures, TLS verification errors, proxy failures, and `Connection refused` instead of collapsing them to a bare `Connection error`, so unreachable endpoints stay distinguishable from unsupported routes or models.
- Provider request normalization should drop whitespace-only multimodal text blocks before strict provider APIs see them, while preserving non-text media blocks.
- Provider `tool_call` ids must stay unique: deduplicate reused ids in streaming responses, and during history sanitization give each `tool_call` within a message a unique normalized id while pairing tool results back through a per-raw-id FIFO, so providers that reuse one id for parallel calls cannot create an ambiguous assistant/tool-result pairing. Non-streaming response parsing should preserve the original upstream `tool_call` id when present (instead of always minting a fresh local id), so log correlation and downstream tool-result linkage stay readable; sanitization will still normalize/dedupe before the next outbound request.
- OpenAI Responses replay must keep input item ids (`msg_*` / `fc_*`) unique within one request through `openai_responses.converters._unique_item_id`; resumed Codex conversations otherwise get rejected for duplicate `rs_*` ids. `call_id` is the tool-result linkage and must stay untouched.
- Direct OpenAI requests for GPT-5 / o1 / o3 / o4 families, or requests with `reasoningEffort`, should prefer the Responses API when talking to real OpenAI endpoints and fall back to Chat Completions only for clear Responses-API compatibility errors.
- The admin visual config should expose `agents.defaults.providerPool.strategy` and `agents.defaults.providerPool.targets` with locale-backed labels/tooltips, a row-based target editor with add/remove/reorder controls, and restart-required badges; leaving all target rows blank should remove the pool from saved config.
- The admin visual config may also expose common `providers.*` credential blocks such as `openrouter`, `openai`, `anthropic`, `deepseek`, `custom`, `ollama`, and `vllm`; group them into locale-backed collapsible provider cards with safe summary chips, mark them clearly as restart-required, and leave less common providers in raw JSON fallback.
- Long tool-heavy turns should compact older tool outputs on demand before the next model call so system prompt memory and recent working context are less likely to fall out of the active context window.
- `memory.user.backend` selects the active user-memory backend. Default `sqlite` keeps the persona's `MEMORY.md` as source of truth and serves top-K BM25 retrieval via a derived FTS5 index at `memory/facts.sqlite`. `file` falls back to whole-file injection.
- When `memory.user.backend=sqlite`, keep the file backend as the conservative prompt-context fallback if SQLite returns empty or the lookup fails. `MemoryRouter` wires this automatically.
- `memory.user.sqlite` exposes `topK`, `maxContextChars`, and `maxFragmentChars` knobs. Values flow through `MemoryRuntimeManager` into `SQLiteUserMemoryBackend`.
- Consolidator wraps every `new_facts` block with a server-controlled `<!-- ts:... tag:... src:turn -->` header before append; LLMs only propose the optional `tag:` token (one of `preference`, `project`, `reference`, `feedback`, `user`, `experience`). `experience` is reserved for distilled task patterns (successful tool sequence for a class of problem), not one-off facts. Dream phase-2 prompt teaches the same format with `src:dream` for fragments it creates.
- `agents.defaults.subagent.models` is a dict of role name → provider/model literal (e.g. `{"fast":"openai/gpt-4.1-mini","strong":"anthropic/claude-opus-4-5"}`). The `spawn` tool now accepts an optional `model` parameter; the server resolves a role name to its configured target, accepts a provider-qualified literal as a pass-through, and falls back to the active default for anything else (logged warning). Empty roles map = current behavior. The role map is hot-reloadable.
- The admin UI exposes a "Subagents" page that lists in-flight background tasks and lets a human-on-the-swarm (HOTS) push a system message into a specific task or cancel it. The message is delivered by `_SubagentHook.before_iteration`, so it lands in front of the next LLM call (not mid-tool-execution). The page is only useful when the runtime forwards `SubagentManager` into `create_http_app` (`hahobot gateway` does this automatically; standalone `create_http_app(...)` callers must pass `subagent_manager=...`).
- HOTS is also reachable from any chat channel via `/jobs` (list), `/jobs inject <id> <message>` (push a system message), and `/jobs cancel <id>` (cancel one task). The handler resolves ownership against `running_tasks_snapshot(session_key=ctx.key)`, so a user can only see and act on subagents spawned from their own session — cross-session interference is blocked at the command layer.
- `channels.xiaoyi` speaks Huawei Xiaoyi's A2A WebSocket protocol (ported from `openJiuwen/jiuwenswarm`, https://atomgit.com/openJiuwen/jiuwenswarm; tracked in `UPSTREAM_PARITY.md`). Hahobot is the client: it opens up to two outbound WebSocket connections (`ws_url1`/`ws_url2`), authenticates with HMAC-SHA256(`sk`, current-millis) carried as the `x-sign` header, sends a `clawd_bot_init` handshake, then keeps the link alive with a 20s app heartbeat. Inbound `message/stream` frames become bus inbound messages with `xiaoyi_session_id`/`xiaoyi_task_id` in metadata; outbound replies are sent as A2A `artifact-update` envelopes wrapped in `agent_response`. A 5s per-session heartbeat prevents the Xiaoyi client from timing out while hahobot is composing a reply.
- The admin visual config should expose `memory.user.backend` plus the three `memory.user.sqlite.*` knobs.
- hahobot does not expose local files over HTTP. If a feature needs a public URL for local files, provide your own static file server and point config such as `mediaBaseUrl` at it.
- Generated screenshots, downloads, and other temporary user-delivery artifacts should be written under `workspace/out`, not the workspace root. Treat that as the generic delivery-artifact root for tools, MCP servers, and skills.
- QQ outbound media can send remote rich-media URLs directly. For local QQ media under `workspace/out`, use direct `file_data` upload only; do not rely on URL fallback for local files. Supported local QQ rich media are images, `.mp4` video, and `.silk` voice.
- QQ outbound send paths should raise on final delivery failure so `ChannelManager` can apply `channels.sendMaxRetries` consistently.
- `channels.voiceReply` currently adds TTS attachments on supported outbound channels such as Telegram, and QQ when the configured TTS endpoint returns `silk`. Supported providers are `openai`, `edge`, and `sovits`. Preserve plain-text fallback when QQ voice requirements are not met.
- Voice replies should follow the active session persona. Build TTS style instructions from the resolved persona's prompt files, and allow optional persona-local overrides from `VOICE.json` under the persona workspace (`<workspace>/VOICE.json` for default, `<workspace>/personas/<name>/VOICE.json` for custom personas).
- `channels.voiceReply.url` may override the TTS endpoint independently of the chat model provider. When omitted, fall back to the active conversation provider URL. Keep `apiBase` accepted as a compatibility alias. Persona `VOICE.json` may also override provider-specific fields such as Edge voice/rate/volume or GPT-SoVITS reference-audio settings.
- `/skill search` queries `https://lightmake.site/api/skills` directly with SkillHub-compatible query params (`page`, `pageSize`, `sortBy`, `order`, `keyword`) and does not require Node.js.
- `/skill derive <name> [brief] [--force]` is the local skill-draft flow. Keep it deterministic and workspace-scoped: derive a reviewable `SKILL.md` draft from the current session / `working_checkpoint`, write under `<workspace>/skills/<slug>/`, and only overwrite an existing skill when the user explicitly passes `--force`.
- Derived skill drafts should seed hahobot-local lifecycle metadata (`triggers`, `tool_tags`, `supersedes`, `last_used`, `success_count`) so later linting and runtime selection can work without hidden side stores.
- `/skill` shells out to `npx clawhub@latest` for `install`, `list`, and `update`; those subcommands still require Node.js/`npx` at runtime.
- Keep ClawHub global options first when shelling out: `--workdir <workspace> --no-input ...`.
- `/skill uninstall` is local workspace cleanup, not a ClawHub subprocess call. Remove `<workspace>/skills/<slug>` and best-effort prune `<workspace>/.clawhub/lock.json`.
- Treat empty `/skill search` output as a user-visible "no results" case rather than a silent success. Surface npm/registry failures directly to the user.
- Never hardcode `~/.hahobot/workspace` for skill installation or lookup. Use the active runtime workspace from config or `--workspace`.
- The implicit default workspace is `<config-dir>/workspace`. If `agents.defaults.workspace` is empty, keep workspace resolution tied to the active config path instead of a separate global default.
- Workspace skills in `<workspace>/skills/` take precedence over built-in skills with the same directory name.
- Built-in skills now include `translate`, `llm-wiki`, `living-together`, `emotional-companion`, and `memorix`. Keep them aligned with the existing skill loader and current built-in tool surface instead of importing NanoMate-only runtime assumptions.
- File memory now also writes structured archive sidecars under `<persona-workspace>/memory/archive/`. Prefer `history_search` / `history_timeline` / `history_expand` for archived conversation recall, and keep `HISTORY.md` grep as a fallback path. `memory.archive.indexBackend="sqlite"` may enable a persona-local SQLite FTS derived cache, but JSONL/chunk sidecars remain the source of truth and `hahobot memory index rebuild` must be able to recreate the cache.
- When Memorix MCP tools are connected, auto-load the built-in `memorix` skill and call `memorix_session_start` with the active workspace as `projectRoot` once per runtime MCP connection and chat session. Keep Memorix scoped to workspace/code memory instead of user-profile memory.
- `hahobot persona import-st-card <file>` imports a SillyTavern character card into `<workspace>/personas/<name>/` by generating `SOUL.md`, `USER.md`, `memory/`, and persona-local metadata under `.hahobot/`.
- `hahobot persona import-st-preset <file> --persona <name>` imports a SillyTavern preset into an existing persona by generating `STYLE.md` plus `.hahobot/st_preset.json`.
- `hahobot persona import-st-worldinfo <file> --persona <name>` imports SillyTavern world info into an existing persona by generating `LORE.md` plus `.hahobot/st_world_info.json`.
- `hahobot companion init [--persona <name>] [--reference-image <path>]` bootstraps a minimal companion persona scaffold. Keep it conservative: create/update only managed companion files (`SOUL.md`, `USER.md`, `STYLE.md`, `VOICE.json`, optional manifest/reference asset copy, optional default heartbeat task), preserve existing files unless `--force` is passed, and never auto-create `PROFILE.md` or `INSIGHTS.md`.
- `hahobot doctor [--json]`, `hahobot model [--json]`, and `hahobot tools [--json]` are read-only runtime inspection commands. Keep them non-destructive and focused on the active config/workspace; `doctor` should summarize readiness, `model` should explain provider resolution / provider-pool routing, and `tools` should summarize web / exec / image-gen / MCP state.
- `hahobot repo status [--json]` and `hahobot repo diff [--staged] [--name-only] [--json]` are read-only repository inspection commands. Keep both the CLI and gateway/chat forms limited to local Git metadata and tracked diff summaries for the active workspace; do not add branch switching, worktree mutation, or commit-writing behavior.
- `hahobot review [--staged] [--base <rev>] [--path <path>] [--json]` is the diff-review command. Keep both the CLI and gateway/chat forms read-only, provider-backed, and findings-first; they should review only the collected Git diff payload for the active workspace and must not write session history, mutate the repo, or silently execute tools.
- `hahobot sessions compact <key> [--json]` and gateway `/compact [key]` should manually trigger the existing session token-consolidation path. Keep them scoped to the target session/workspace, persist `last_consolidated` updates where applicable, and avoid inventing a second memory pipeline or unrelated workspace writes.
- `hahobot.agent.hook_bridge.ExternalHookBridge` is the supported SDK-side bridge from `AgentHook` lifecycle events to external commands. Keep the stdin payload JSON-based with the current `schema_version/event/context` shape, default event selection non-streaming, and reserve `{"continue": false}` / exit code `2` for explicit pre-run or pre-tool blocking.
- `tools.exec.allowedEnvKeys` is the narrow allowlist for passing parent env vars into shell subprocesses. Keep the default isolated env, and only forward explicitly named keys such as `JAVA_HOME` / `GOPATH` when configured. POSIX `tools.exec.pathAppend` must stay env-backed through `HAHOBOT_PATH_APPEND` rather than interpolating raw config into shell syntax. Shell subprocesses must be spawned with stdin detached (`stdin=asyncio.subprocess.DEVNULL`) so a command that reads from stdin sees EOF immediately instead of hanging on the inherited terminal.
- `hahobot agent --continue` should resume the most recent local CLI session without making the user name it manually, and `hahobot sessions list [--json]` should expose recent saved sessions for inspection. Keep both read-only with respect to existing session history until the user actually sends a new message.
- `hahobot sessions export <key> [--format md|json] [--output <path>]` should export an existing saved session to a local artifact without mutating the underlying session history. Default output should live under `workspace/out/sessions/`.
- `hahobot agent --pick-session` should prompt from recent local CLI sessions and then continue using the selected session key. Keep it mutually exclusive with explicit `--session` and `--continue`.
- `hahobot agent --multiline` should only affect interactive CLI input mode. Keep the default single-line behavior unchanged, and make the submit gesture explicit in the UI (`Enter` newline, `Ctrl+J` submit).
- Interactive CLI input and history writes should sanitize malformed Unicode surrogate code points before dispatch/persistence while preserving valid emoji and Unicode text.
- Interactive `hahobot agent` input should provide slash-command completion for built-in commands and common subcommands, including workspace-derived persona / scene names and the CLI-local `/session ...` / `/repo ...` / `/review ...` / `/compact` controls, without affecting normal free-form chat input.
- Local interactive `/session ...` commands inside `hahobot agent` should still be handled by the CLI shell itself rather than forwarded to the model. Keep `/session current`, `/session list`, `/session show [key]`, `/session export [key]`, `/session use <key>`, and `/session new [name]` scoped to local CLI session management there, while the equivalent gateway/chat commands stay available through AgentLoop. In gateway mode, `/session use <key>` and `/session new [name]` should reroute only the current chat, and `/session use default` should clear that reroute.
- Local interactive `/repo ...` commands inside `hahobot agent` should also stay in the CLI shell. Keep `/repo status`, `/repo diff`, and `/repo diff staged` read-only there, while the same command family is also available through gateway/chat routing.
- Local interactive `/review` commands inside `hahobot agent` should also stay in the CLI shell. Keep `/review` and `/review staged` scoped to provider-backed local diff review there, while the same review flow is also available through gateway/chat routing.
- Local interactive `/compact` commands inside `hahobot agent` should also stay in the CLI shell. Keep `/compact` and `/compact <key>` scoped to the existing token-consolidation implementation there, while the equivalent gateway/chat command remains available through AgentLoop and returns plain-text status.
- Provider requests through `AgentRunner` should keep a finite timeout by default (`HAHOBOT_LLM_TIMEOUT_S`, legacy `NANOBOT_LLM_TIMEOUT_S`, `0` disables) so hung gateways do not starve per-session locks.
- Streaming provider httpx clients (anthropic, openai_compat, openai_codex) share `LLMProvider._stream_idle_timeout_s()`, which reads `HAHOBOT_STREAM_IDLE_TIMEOUT_S` (default `90`). Do not reintroduce a `NANOBOT_*` fallback for this knob — it is a fresh hahobot-only env var, not part of the legacy rename-transition allowlist (`HAHOBOT_PATH_APPEND`, `HAHOBOT_LLM_TIMEOUT_S`).
- Prompt history may use `Session.get_history(..., include_timestamps=True)` to expose persisted user/assistant turn timestamps as `[Message Time: ...]` anchors; do not mutate the underlying session message content for this view.
- Proactive deliveries from cross-session `message` tool sends, cron, and heartbeat should be recorded into the target `channel:chat` session for reply continuity.
- `hahobot sessions show <key> [--json]` should expose the saved metadata, any persisted `working_checkpoint`, and a bounded tail of recent messages for one session, without implicitly creating new sessions when the requested key does not exist.
- `hahobot companion doctor [--persona <name>] [--json]` is the read-only readiness check for companion workflows. Keep it inspecting the current workspace/config without writing files; it should cover persona files, heartbeat tasks, voice reply readiness, image generation, reference images, and enabled channels.
- Persona workspaces may include optional `PROFILE.md`, `INSIGHTS.md`, `STYLE.md`, and `LORE.md`. `PROFILE.md` is the user-model layer for stable user facts/preferences, `INSIGHTS.md` is the learned collaboration-guidance layer for proven workflows and pitfalls, and both should load as separate sections; `USER.md` remains relationship framing; `STYLE.md` and `LORE.md` should continue loading after `SOUL.md` and `USER.md`.
- Workspace template sync should not auto-create `PROFILE.md` or `INSIGHTS.md`; they remain optional overlays. Keep the seeded `USER.md` template aligned with its relationship-framing role instead of using it as a user-profile questionnaire.
- Dream memory hygiene should prefer one canonical bullet per remembered fact/pattern. Conflicting `PROFILE.md` / `INSIGHTS.md` bullets should be replaced rather than accumulated. For tentative or revalidated profile/insight bullets, prefer structured metadata comments such as `<!-- hahobot-meta: confidence=low -->` and `last_verified=YYYY-MM-DD`; legacy `(verify)` markers remain compatibility-only.
- Persona-local `.hahobot/st_manifest.json` may define `response_filter_tags`. Apply those tags only to final user-visible output; preserve the unfiltered assistant content in saved session history.
- Persona-local `.hahobot/st_manifest.json` may also define `reference_image` and `reference_images`. When `tools.imageGen.enabled` is true, the `image_gen` tool should resolve `__default__` and `__default__:scene` against that manifest and save generated output under `workspace/out/image_gen`.
- The local WhatsApp bridge should honor standard proxy environment variables (`https_proxy`, `http_proxy`, `all_proxy`, including SOCKS5 URLs) because bridge deployments may run behind proxies.

## Multi-Instance Channel Notes
The repository supports multi-instance channel configs through `channels.<name>.instances`. Each
instance must define a unique `name`, and runtime routing uses `channel/name` rather than
`channel:name`.

- Supported multi-instance channels currently include `whatsapp`, `telegram`, `discord`,
  `feishu`, `mochat`, `dingtalk`, `slack`, `email`, `qq`, `matrix`, and `wecom`.
- Keep backward compatibility with single-instance configs when touching channel schema or docs.
- If a channel persists local runtime state, isolate it per instance instead of sharing one global
  directory.
- `matrix` instances should keep separate sync/encryption stores.
- `mochat` instances should keep separate cursor/runtime state.
- `whatsapp` multi-instance means multiple bridge processes, usually with different `bridgeUrl`,
  `BRIDGE_PORT`, and `AUTH_DIR` values.
